whl包重命名方法

动机

目前产品组件中，所需依赖基本以Python wheel包 格式为主。有时因为产品需求，需要重新编译或使用与官方whl 不同的版本，但此时会因为名字相同而冲突。一种比较取巧的方式是，上传wheel之前，改掉package 的命名，使得pypi索引时名字可以唯一，requirements中可以直接使用自己命名的特定版本。

重命名方式

编译重命名

该方法是原先linux使用的方式，因为linux中修改了元信息的name，再压缩为whl会失效。所以需要修改setup.py中的name，然后走编译运行的流程。最终用twine upload上whl包。

此外，因为pypi上只允许上传manylinux标记的whl包，因此若whl不满足，就必须自己编译出manylinux才能上传。

免编译重命名

windows在不需要编译特定功能时，可以直接使用压缩包重新打包成whl格式。

下面是linux中试验通过的免编译重命名流程：

  • 通过wheel unpack *.whl命令 解压后，
  • 修改METADATA中的Name字段，并把文件夹xxx-yyy.dist-info（xxx原包名，yyy是版本号）的原包名也改成METADATA中的新包名，
  • 用wheel pack 文件夹名称压缩回whl。此时的whl名称应该自动变为 所需要的新名字。

whl包的检查

当whl包不是manylinux，无法上传到pypi上，此时可尝试用

auditwheel repair *.whl，(最好应该在manylinux容器里修复：docker run -i -t -vpwd:/io quay.io/pypa/manylinux2010_x86_64:2021-04-05-b4fd19d /bin/bash) 。若得到错误：

cannot repair "...." to "manylinux1_x86_64" ABI because of the presence of too-recent versioned symbols. You'll need to compile the wheel on an older toolchain

一般需要重新编译才能解决问题，可以用 auditwheel-symbols --manylinux 2010 *.whl （用pip install auditwheel-symbols）确认是哪个符号使用了不兼容的glibc版本库。

pypi包上传错误

rst文档错误导致

若上传或测试whl包出现错误，观察错误信息中，是否是readme描述信息所导致。该问题很大可能是因为rst语法问题导致。 README.rst说明文档在 github 上正常解析，但上传时 PyPi 中解析错误，这是由于 github 和 PyPi 对 reStructuredText 文档解析工具的版本不同，PyPi 使用的是 0.8 版本，而 github 使用的是 0.9 或 0.10。 PyPi 对文档要求很严格， 语法错误是导致不能解析的主要原因。查找解决办法就是查看文档哪里的语法不对，但文档太长只有借助工具检测：

xxxxxxxxxx
2
1
pip install collective.checkdocs
2
python setup.py checkdocs

确认为rst引起的错误后，最简单的处理方法是将setup.py里的long description设置为None。 因为包的长描述信息一般都直接使用readme解析生成。
